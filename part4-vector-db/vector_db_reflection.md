## Vector DB Use Case

A traditional keyword-based database search would not be sufficient for a law firm that wants to search 500-page contracts using plain English questions such as “What are the termination clauses?”. Keyword search relies on exact or near-exact word matches, which means it often fails when the wording of the query differs from the wording used in the document. Legal contracts frequently use complex language, synonyms, and long sentences, making it difficult for keyword-based systems to capture the true intent of the query.

A vector database enables semantic search by storing embeddings that represent the meaning of text rather than just the words themselves. When contracts are broken into clauses or sections and converted into vector embeddings, a lawyer’s natural-language question can also be embedded into the same vector space. The system can then retrieve the most semantically similar sections of the contract, even if the exact keywords “termination clause” are not present. For example, phrases like “agreement may be ended by either party” would still be correctly matched.

In this system, the vector database plays a critical role in efficiently storing and searching high-dimensional embeddings at scale. It allows fast similarity searches across millions of text chunks, which would be impractical using a traditional relational database. The vector database would typically work alongside a document store and possibly a large language model, enabling accurate clause retrieval and answer generation. Overall, vector databases make natural-language contract search practical, scalable, and far more effective than keyword-based approaches.
